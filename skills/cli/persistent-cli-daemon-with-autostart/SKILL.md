# Persistent CLI with Auto-Started Daemon

## When to use

- Your CLI does something stateful (browser session, DB connection, REPL) that's expensive to start.
- You want `mycli navigate URL; mycli click uid` to share state across invocations but still feel like normal one-shot commands.
- You want a managed lifecycle: explicit `start`/`stop`/`status` verbs for ops, implicit start for convenience.

## How it works

- Define three lifecycle subcommands: `start [--flags...]` forwards flags to the daemon startup, `stop` signals the daemon to exit, `status` queries the daemon and prints PID/socket/start date/version/args.
- Auto-generate all other subcommands from your tool schema (see `generate-cli-from-mcp-tool-list` pattern). Each auto-command's handler is:
  1. If daemon not running, call `start([])` internally.
  2. `sendCommand({method: 'invoke_tool', tool: name, args})` to the socket.
  3. Print the response as Markdown by default, JSON when `--output-format=json`.
- On `start`, if another daemon is already running, `stopDaemon()` first then start fresh — the user's intent is "bring up a new daemon with these args".
- Set `process.title = 'mycli'` early so ps/top can tell the daemon from the client.
- Default some implicit flags (`--viaCli`, structured content on) that the CLI always wants but the MCP server doesn't default.

## Example

```ts
process.title = 'chrome-devtools';
const y = yargs(hideBin(process.argv))
  .scriptName('chrome-devtools')
  .demandCommand()
  .strict();

y.command('start', 'Start or restart the daemon', y => y.options(startCliOptions), async argv => {
  if (isDaemonRunning()) await stopDaemon();
  await startDaemon([...serializeArgs(cliOptions, argv), '--viaCli']);
});
y.command('status', 'Check daemon state', async () => {
  if (!isDaemonRunning()) return console.log('daemon is not running');
  const r = await sendCommand({method:'status'});
  console.log(r.result);
});
y.command('stop', 'Stop the daemon', async () => { if (isDaemonRunning()) await stopDaemon(); });

for (const [name, def] of Object.entries(autoGeneratedCommands)) {
  y.command(buildCommandString(name, def), def.description, buildBuilder(def), async argv => {
    if (!isDaemonRunning()) await startDaemon([]);
    const r = await sendCommand({method:'invoke_tool', tool: name, args: extractArgs(argv, def)});
    console.log(await handleResponse(r.result, argv['output-format']));
  });
}
await y.parse();
```

## Gotchas

- Auto-start on an unknown tool name should still fail fast with a helpful error, not silently start a daemon and then send garbage. Validate the command name against your generated list before the `start([])` call.
- Explicit `start` should *always* restart (not no-op on "already running"). Users who ran `mycli start` expect their new flags to apply.
- `stop` needs a timeout wrapped around `waitForFile(pidFilePath, removed=true)` — if the daemon crashed mid-stop it may never delete the PID file.
- If you add `--output-format=json`, switch the response serialization to raw JSON so scripts can parse it. Default human output should be Markdown with file paths inlined.
- Document that the daemon persists across CLI invocations — user confusion "why didn't my new tab appear" usually means a stale daemon with old state.
