# bff-pattern-data-simulation

The BFF dashboard simulates per-BFF latency metrics using a bounded random walk. Each BFF channel (mobile, web, IoT) maintains a rolling history array of 40 data points. On each 1-second tick, a new value is generated by adding a uniform random delta (±12ms) to the previous value, clamped to a minimum of 5ms. This produces realistic-looking latency curves that drift and occasionally spike without requiring a real backend. The rolling window is enforced by shifting old values off the front once the array exceeds `maxPts`, keeping memory constant.

Per-BFF dashboard cards display three derived metrics: the arithmetic mean of the window (`Avg`), the maximum value (`Peak`), and a randomized RPM counter (200–900) to simulate throughput variance. Each card renders its history as a filled area chart on a `<canvas>` element, using the device-pixel-ratio doubling trick (`canvas.width = clientWidth * 2`) for crisp HiDPI rendering. The fill is created by closing the line path to the bottom edge and applying the tier color at 8% opacity, giving a subtle gradient-under-curve effect. An aggregate card spans all columns and sums the three BFF histories point-by-point to show total system throughput.

This simulation pattern is reusable for any multi-channel BFF monitoring scenario: initialize N history arrays, tick with a random walk, derive summary stats, and render sparkline charts. The key design choice is keeping simulation logic (random walk + rolling window) completely separate from rendering (canvas draw calls), which lets the same data model drive different visualization types — the flow app's particle system and the dashboard's area charts both ultimately consume the same kind of per-BFF time-series structure.
