# Skill Reactor

> **Why.** `skill-multiplier` is stochastic \u2014 fun, but the children feel arbitrary. `skill-breeder` is hand-curated \u2014 precise, but slow. Reactor sits between: you declare **reaction rules** (e.g. `category=kafka + tag=tenant \u2192 new skill`), and the engine fires them against the entire pool every tick. Newly produced children re-enter the pool and can satisfy other rules, which is how the propagation stays infinite without becoming noise. Think Conway's Game of Life, but for skills.

## Features

- **Rule editor** \u2014 add/remove/edit rules inline; persisted to `localStorage` between sessions.
- Four **pattern kinds**: `category=X`, `tag=Y`, `regex:...`, or a substring fallback. Case-insensitive.
- **Step** (one tick) or **Run** (continuous every 250ms). 3-tick cooldown per unique `(A, B, rule)` triple prevents instant re-firing.
- **Per-rule fire counter** so you can see which rules are productive vs dormant.
- Labeled children: each child's name is prefixed by the firing rule's label (`"multi-tenant kafka: ..."`), making reactor output easy to distinguish from other generators.
- **Default starter ruleset** loaded on first run (kafka\u00d7tenant, mongo\u00d7time-series, cache\u00d7ttl, testing\u00d7backend).
- **Export** dumps rules + per-rule stats + all children.
- Same zero-dep `synth.mjs` engine \u2014 no LLM.

## Usage

```bash
cd /path/to/skills-hub
node /path/to/skill-reactor/browser/build-index.mjs

cd skill-reactor/browser
CORPUS=/path/to/skills.json node serve.mjs
# \u25b6 skill reactor \u2192 http://localhost:4179/
```

Corpus resolution: `$CORPUS` \u2192 `./skills.json` \u2192 `../../browser/skills.json`.
