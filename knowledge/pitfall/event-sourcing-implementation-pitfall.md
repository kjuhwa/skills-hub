# event-sourcing-implementation-pitfall

**Mutable accumulator contamination during fold.** The most dangerous event-sourcing bug is mutating the accumulator object inside the reducer instead of producing a fresh copy per step. When the fold function does `s.items[key] = newQty` directly on the accumulator, time-travel appears to work on forward scrub but corrupts on backward scrub — prior snapshots silently reflect future state because they share object references. This is invisible in normal append-only operation and only surfaces when a user scrubs the replay slider backward and sees data that shouldn't exist yet. The fix is returning a new object from every reducer branch (`{ ...s, items: { ...s.items, [key]: newQty } }`), and the time-travel slider itself is the best regression test: if scrubbing backward ever shows values from a later version, a mutation leak exists. In the analyzed apps, the timeline app's reducer mutates properties directly on a fresh-per-call accumulator, which is safe only because `project()` creates a new `s` on every invocation — but if the caller ever caches and reuses the returned state object, the bug appears.

**Silent event loss from incomplete reducer coverage.** When a reducer's `if`/`switch` chain doesn't handle a newly introduced event type, the event is silently dropped — no error, no log, no indication that the materialized state has diverged from the event log. This becomes a split-brain in CQRS architectures: the read-model projection may handle the new event type correctly while the aggregate reducer ignores it, so the write model and read models drift apart without any observable error. Every reducer should include an explicit `default` or `else` branch that at minimum logs an unknown-event warning. Integration tests should assert that the set of types the reducer handles matches the set of types the command side can emit, catching coverage gaps at build time rather than in production.

**Snapshot boundary mismanagement and double-counting.** The ledger app implements a snapshot mechanism that stores `{ balance, eventCount }` as a checkpoint, intending to let the fold start from the snapshot instead of replaying from event zero. However, the fold function never actually consumes the snapshot — it always replays the entire log, making the snapshot dead code. The deeper pitfall is that if a snapshot records "balance after event N" but the fold replays from event 0 without skipping events 0–N, every pre-snapshot event is counted twice. Conversely, if the fold skips to event N+1 but the snapshot was taken at a slightly different boundary (e.g., off-by-one on the event index), one event is either lost or doubled. The safe pattern is: store the snapshot with the exact event sequence number it includes, and have the fold accept an optional `(startState, fromIndex)` tuple, with an assertion that `events[fromIndex - 1]` matches the snapshot's sequence number. Without this correlation guard, snapshots become a source of silent balance drift rather than an optimization.
