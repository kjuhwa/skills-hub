TOML parses section headers left-to-right. A `[foo]` header means every subsequent key belongs to `foo` until the next header. So when writing a managed block into a user-owned config.toml:

- If the managed block opens with `[sandbox_workspace_write]`, any user content that happened to sit below the block gets reparented into that table.
- If the managed block appends to a file that already ends inside a user table (e.g. `[permissions.multica]`), a bare `sandbox_mode = "..."` key gets parsed as `permissions.multica.sandbox_mode` and the real setting is silently ignored.

Correct form: top-level dotted-key assignments, placed at the top of the file, delimited by markers:

```toml
# BEGIN multica-managed (do not edit; regenerated by daemon)
sandbox_mode = "workspace-write"
sandbox_workspace_write.network_access = true
# END multica-managed
```

Rewrite idempotently on every task: strip the existing block (by markers), prepend the freshly rendered one.

## Why

Either trap is silent — the config file still parses successfully, it just parses to a shape you didn't intend. Without dotted-key discipline, you end up debugging "why doesn't my sandbox policy take effect" and find it 48 hours later.
