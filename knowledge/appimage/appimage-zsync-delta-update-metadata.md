# AppImage zsync Delta Update Metadata

## Context

AppImage supports optional binary delta updates via the `zsync` protocol: rather
than downloading the full new AppImage on every update, the updater downloads
only the changed blocks (typically 1-5% of the file size for minor upstream
updates). This requires the AppImage to embed an update URL and a corresponding
`.zsync` metadata file to be hosted alongside the AppImage.

Users who build AppImages locally (from source or with a local installer) want
to understand whether their self-built AppImage can receive automatic updates.

## Observation

For `zsync` delta updates to work, three things must be true:

1. The AppImage binary must embed an `update` field (in the type 2 AppImage
   header section 5) pointing to the `.zsync` file URL. Tools like
   `appimagetool` or `linuxdeployqt` embed this when building with
   `--updateinformation zsync|<url>`.

2. A `.zsync` file must exist at the embedded URL. This file contains block
   checksums of the current AppImage and is generated by the `zsyncmake` tool.

3. The AppImage update client (AppImageUpdate, Gear Lever's update feature)
   reads the embedded URL, downloads the `.zsync` file, computes block diffs,
   and downloads only the changed blocks.

Locally-built AppImages have none of this: the `appimagetool` invocation in a
typical build script does not pass `--updateinformation`, so no URL is embedded.
No `.zsync` file is generated or hosted.

GitHub Releases-hosted AppImages (built by the official CI pipeline) include
the embedded URL pointing to a `.zsync` file published alongside the `.AppImage`
release asset. These receive efficient delta updates automatically through
compatible tools.

## Why it happens

The `zsync` workflow requires infrastructure: a stable URL for the `.zsync`
file, a hosting location (GitHub Releases, a CDN), and the CI to generate and
upload the `.zsync` file on each release. This infrastructure is typically only
set up for official releases, not local builds.

The `.zsync` file must be regenerated for each new AppImage release — it
encodes the checksums of the specific binary. A stale `.zsync` from an older
release will not help update to a newer one.

## Practical implication

For projects that publish AppImages, add `.zsync` generation to the CI release
pipeline:

```bash
# After building the AppImage:
zsyncmake -C -u "https://releases.example.com/app-latest.AppImage" \
  -o "app-latest.AppImage.zsync" \
  "app-latest.AppImage"

# Rebuild AppImage with embedded update URL:
appimagetool \
  --updateinformation "zsync|https://releases.example.com/app-latest.AppImage.zsync" \
  AppDir/ app-latest.AppImage
```

Inform users of locally-built AppImages that:
- The `.zsync` file is unused/irrelevant for their build.
- They can delete `.zsync` files without consequence.
- To get delta updates, they should use the GitHub Releases AppImage or switch
  to the `.deb`/`.rpm` package format.
- Desktop integration tools (Gear Lever) can still manage manually-configured
  updates for local AppImages, but delta updates require the hosted `.zsync`.
